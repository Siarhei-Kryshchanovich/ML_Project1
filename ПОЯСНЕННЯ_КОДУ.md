# ПОЯСНЕННЯ_КОДУ

## 1. Короткий огляд проєкту
Цей проєкт запускає відтворювані експерименти з виявлення шахрайства на двох датасетах (`creditcard.csv` і PaySim), порівнює кілька ML-моделей з різними стратегіями роботи з дисбалансом класів, підбирає поріг за валідаційними даними та зберігає метрики/графіки для звіту.

## 2. Потік виконання (головна частина)

Нижче — реальний порядок виконання, коли ви запускаєте:

```bash
python main.py
```

1. **Python запускає `main.py` і імпортує модулі.**  
   `main.py` імпортує константи з `config.py`, завантаження даних із `data_loader.py`, препроцесинг із `preprocessing.py`, логіку моделей/пайплайнів із `models.py`, оцінювання з `evaluation.py` та утиліти з `utils.py`.

2. **Блок `__main__` запускає експерименти.**  
   В кінці `main.py` викликається `run_all_experiments(include_is_flagged_fraud=False)`.

3. **Створюються директорії для результатів.**  
   Усередині `run_all_experiments` виклики `ensure_dir(OUTPUT_DIR)` і `ensure_dir(PLOTS_DIR)` створюють `outputs/` та `outputs/plots/`, якщо їх ще немає.

4. **Готуються базові моделі.**  
   `get_base_models()` формує словник моделей:
   - Logistic Regression
   - Decision Tree
   - Random Forest
   - XGBoost (лише якщо імпорт у `models.py` вдався)

5. **Перевіряється опційна поведінка XGBoost.**  
   Якщо xgboost не встановлений, `xgboost` відсутній у `base_models`, `main.py` друкує інформаційне повідомлення і пропускає цю модель.

6. **Фільтрується список моделей.**  
   `MODEL_NAMES` із `config.py` фільтрується так, щоб залишити лише реально доступні моделі.

7. **Починається цикл по датасетах.**  
   `for dataset_name in DATASETS:` проходить по налаштованих датасетах (`creditcard`, `paysim`) із `config.py`.

8. **Кожен датасет завантажується через `load_dataset`.**  
   `load_dataset(dataset_name, include_is_flagged_fraud=False)`:
   - читає CSV з `DATASETS[dataset_name]["path"]`
   - бере цільову колонку з `DATASETS[dataset_name]["target"]`
   - перетворює target у цілі числа
   - формує список колонок для видалення: `drop_cols` + target
   - для PaySim додатково прибирає `isFlaggedFraud` за замовчуванням
   - повертає `X` (ознаки) і `y` (ціль)

9. **Вибираються target-колонки.**  
   - Для credit card: `Class`
   - Для PaySim: `isFraud`

10. **Видаляються зайві колонки.**  
    - Credit card: видаляється лише target (додаткових колонок у config немає)
    - PaySim: видаляються `nameOrig`, `nameDest`, target (`isFraud`) і зазвичай `isFlaggedFraud`

11. **Розбиття на train/validation/test.**  
    У `main.py` використано двоетапне стратифіковане розбиття:
    - Крок A: `train_test_split(..., test_size=TEST_SIZE)` -> `train_temp` і `test`
    - Крок B: `train_test_split(..., test_size=VAL_SIZE)` на `train_temp` -> `train` і `val`

    З поточними параметрами (`TEST_SIZE=0.20`, `VAL_SIZE=0.25` від залишку) фактично маємо:
    - ~60% train
    - ~20% validation
    - ~20% test

12. **Починається цикл по моделях.**  
    Для кожної доступної моделі зі списку `model_names`.

13. **Починається цикл по стратегіях балансування.**  
    Для кожної стратегії з `BALANCING_STRATEGIES`:
    - `baseline`
    - `class_weight`
    - `undersample`
    - `oversample`
    - `smote`
    - `adasyn`

14. **Формується назва експерименту.**  
    `experiment_name(dataset_name, model_name, strategy)` створює, наприклад, `paysim__random_forest__smote`.

15. **Будується препроцесинг (`build_preprocessor`).**  
    `build_preprocessor(dataset_name, X_train)`:
    - визначає числові колонки за dtype
    - визначає категоріальні колонки за dtype
    - створює numeric pipeline: імпутація медіаною + стандартизація

16. **Деталі обробки числових і категоріальних ознак.**  
    - **Числові ознаки:** завжди проходять через імпутер + scaler.
    - **Категоріальні ознаки:**
      - Для PaySim, якщо є `type`, для неї створюється окрема гілка (most-frequent імпутація + one-hot encoding).
      - Інші категоріальні колонки (якщо є) обробляються fallback-пайплайном (також імпутація + one-hot).

17. **Будується training pipeline (`build_pipeline`).**  
    `build_pipeline(...)` створює imbalanced-learn pipeline з кроками:
    1) preprocessor  
    2) sampler (лише для стратегій ресемплінгу)  
    3) model

18. **Логіка вибору стратегії балансування.**  
    - `baseline`: без sampler, базові параметри моделі
    - `class_weight`: без sampler; ваги класів задаються в `_apply_class_weight`:
      - sklearn-моделі -> `class_weight="balanced"`
      - xgboost -> `scale_pos_weight = negatives/positives`
    - `undersample` -> `RandomUnderSampler`
    - `oversample` -> `RandomOverSampler`
    - `smote` -> `SMOTE`
    - `adasyn` -> `ADASYN`

19. **Відбувається навчання моделі.**  
    `pipeline.fit(X_train, y_train)` виконує препроцесинг + (за потреби) ресемплінг + тренування моделі на train.

20. **Обчислюються валідаційні ймовірності.**  
    `val_prob = pipeline.predict_proba(X_val)[:, 1]` — ймовірність класу шахрайства (`1`) на validation.

21. **Підбір порогу на validation.**  
    `tune_threshold(y_val, val_prob, cost_fp, cost_fn)`:
    - перебирає пороги від `0.01` до `0.99`
    - рахує метрики та cost для кожного порогу
    - вибирає поріг із мінімальним expected cost
    - повертає найкращий поріг і повну таблицю порогів

22. **Таблиця порогів зберігається в CSV.**  
    Зберігається у файл:  
    `outputs/<experiment_name>__thresholds.csv`

23. **Обчислюються фінальні ймовірності на test.**  
    `test_prob = pipeline.predict_proba(X_test)[:, 1]` на відкладеному тестовому наборі.

24. **Фінальне оцінювання на test у двох режимах порогу.**  
    `compute_metrics(...)` викликається двічі:
    - для порогу `0.5`
    - для підібраного порогу з validation

25. **Як саме рахуються метрики.**  
    У `evaluation.py` рахуються:
    - ROC-AUC (`roc_auc_score`)
    - PR-AUC (`average_precision_score`)
    - recall, precision, F1, accuracy
    - елементи confusion matrix (`tn, fp, fn, tp`)
    - expected cost = `fp * COST_FP + fn * COST_FN`

26. **Будуються і зберігаються графіки.**  
    - `save_curves(...)` зберігає ROC і PR криві:
      - `outputs/plots/<experiment_name>_roc.png`
      - `outputs/plots/<experiment_name>_pr.png`
    - `save_confusion_matrix(...)` зберігає confusion matrix для tuned threshold:
      - `outputs/plots/<experiment_name>__cm_tuned.png`

27. **Запускається крос-валідація на train_temp.**  
    Заново будуються препроцесор/пайплайн на `X_train_temp, y_train_temp`, потім `cross_validate` з `StratifiedKFold(CV_FOLDS)` рахує CV-метрики (mean/std).

28. **Додається один рядок результатів.**  
    У `results_rows` додається словник із dataset/model/strategy, tuned threshold, test-метриками, cost при 0.5 і CV-підсумками.

29. **Внутрішні цикли продовжуються.**  
    Потік повторюється для наступної стратегії, потім наступної моделі, потім наступного датасету.

30. **Формується і зберігається загальна порівняльна таблиця.**  
    Після завершення всіх циклів `results_rows` перетворюється у DataFrame, сортується і зберігається у:
    `outputs/comparison_table.csv`

## 3. Ролі допоміжних файлів
- **`main.py`**: центральна оркестрація; цикли, розбиття даних, навчання/оцінювання, збереження результатів.
- **`config.py`**: усі константи експерименту (шляхи, датасети, вартості помилок, моделі, стратегії, split, CV).
- **`data_loader.py`**: завантаження датасетів, виділення target, логіка видалення колонок.
- **`preprocessing.py`**: визначення типів ознак і побудова `ColumnTransformer`.
- **`models.py`**: фабрика моделей, фабрика sampler-ів, налаштування class weight, побудова pipeline.
- **`evaluation.py`**: обчислення метрик, підбір порогу, побудова ROC/PR і confusion matrix.
- **`utils.py`**: утиліти для створення директорій і формування назв експериментів.
- **`requirements.txt`**: список Python-залежностей.
- **`README.md`**: коротка інструкція з запуску та опису проєкту.

## 4. Логіка для конкретних датасетів

### `creditcard.csv`
- Завантажується з `data/creditcard.csv`.
- Target-колонка: `Class`.
- Додаткових колонок для видалення в config немає.
- Основний потік: числовий препроцесинг + вибрана стратегія балансування/модель.

### PaySim (`PS_20174392719_1491204439457_log.csv`)
- Завантажується з шляху PaySim у `config.py`.
- Target-колонка: `isFraud`.
- Завжди видаляються `nameOrig`, `nameDest` (ідентифікатори).
- За замовчуванням видаляється `isFlaggedFraud` (`include_is_flagged_fraud=False`) для зменшення ризику leakage.
- `type` обробляється як категоріальна ознака з one-hot encoding.

### Колонки, які ви попросили пояснити
- **`type`**: категоріальна ознака типу транзакції, кодується через OHE у препроцесингу.
- **`nameOrig`**: видаляється в loader як ідентифікатор.
- **`nameDest`**: видаляється в loader як ідентифікатор.
- **`isFlaggedFraud`**: за замовчуванням виключається; включити можна лише якщо явно передати `include_is_flagged_fraud=True`.

## 5. Вихідні файли
Артефакти зберігаються в `outputs/`:

1. **Таблиця порогів для кожного експерименту (CSV)**  
   `outputs/<dataset>__<model>__<strategy>__thresholds.csv`  
   Містить метрики/вартість для порогів 0.01..0.99 і вибраний найкращий поріг.

2. **Графіки для кожного експерименту** в `outputs/plots/`:
   - ROC curve (PNG)
   - PR curve (PNG)
   - confusion matrix (PNG) для tuned threshold

3. **Загальна порівняльна таблиця**  
   `outputs/comparison_table.csv`  
   Містить по одному рядку на комбінацію dataset/model/strategy з test- і CV-метриками.

Як це використати у Report 2:
- `comparison_table.csv` — для порівняння моделей і стратегій,
- `__thresholds.csv` — для обґрунтування вибору порогу,
- ROC/PR/CM графіки — як візуальні докази якості та компромісів.

## 6. Практичні примітки
- **Запобігання leakage:** препроцесинг і ресемплінг вбудовані в pipeline та навчаються лише на train/fold; `isFlaggedFraud` у PaySim вимкнено за замовчуванням.
- **Коректний підбір порогу:** поріг налаштовується тільки на validation, а на test застосовується один раз.
- **Робота з дисбалансом:** стратегія задається явно в циклах, тому порівняння коректне.
- **Опційний XGBoost:** якщо бібліотеки немає, експерименти для інших моделей все одно виконуються.
- **Відтворюваність:** `RANDOM_STATE` з `config.py` передається у split, моделі та sampler-и.
